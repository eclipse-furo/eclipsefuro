This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

**eclipsefuro** (フロー Furo) is a protobuf-first framework. It uses Protocol Buffers as the sole Interface Definition Language (IDL) and provides protoc plugins that generate TypeScript code and JSON Schema files from `.proto` definitions.

## Repository Structure

This is a **multi-module monorepo** with independent Go modules (each with their own `go.mod` and vendored dependencies):

| Directory | Language | Description |
|-----------|----------|-------------|
| `protoc-gen-open-models/` | Go | Protoc plugin generating TypeScript for `@furo/open-models` (Literal/Transport/Model types, enums, services) |
| `protoc-gen-om-jsonschema/` | Go | Protoc plugin generating JSON Schema (Draft 2020-12) from proto messages |
| `protoc-gen-debugfile/` | Go | Protoc plugin that captures the raw `CodeGeneratorRequest` to a file for replay-based debugging |
| `BEC/` | Docker | Build Essentials Container — Docker image bundling protoc, Go, Node, and all furo tooling |
| `docs/` | Hugo | Legacy documentation sites (furo, fidl, fbp, furoc, web) |

Each plugin subdirectory is a self-contained Go project. Always `cd` into the plugin directory before running Go commands.

## Build Commands

### protoc-gen-open-models
```bash
cd protoc-gen-open-models
go build -o protoc-gen-open-models .
```
No tests in this module. Tests live in the companion web project: `eclipse-furo/eclipsefuro-web` (packages/furo-open-models/test).

**Debug mode**: Use `--replay-request=<path>` to replay a previously captured request (from `protoc-gen-debugfile`) without needing protoc.

### protoc-gen-debugfile
```bash
cd protoc-gen-debugfile
go build -o protoc-gen-debugfile .
```

Captures the raw `CodeGeneratorRequest` for replay-based debugging. Usage:
```bash
protoc \
  --debugfile_out=. \
  --debugfile_opt=/tmp/request.bin \
  --om-jsonschema_out=./schema \
  -I./proto_dependencies -I./proto \
  $(find proto -iname "*.proto")
```

### protoc-gen-om-jsonschema
```bash
cd protoc-gen-om-jsonschema
go build -o protoc-gen-om-jsonschema .

# Run all tests
go test -v ./...

# Run tests with coverage
go test -coverprofile=coverage.out ./... && go tool cover -func=coverage.out

# Generate schemas (for manual verification / regenerating test fixtures)
./test-gen-open-models.sh          # default options
./test-gen-open-models-strict.sh   # strict_any + strict_map + strict_oneof
./test-gen-open-models-exclude.sh  # exclude_packages / exclude_messages
```

**Debugging**: Use `protoc-gen-debugfile` to capture the CodeGeneratorRequest, then run standalone with `./protoc-gen-om-jsonschema --replay-request=/tmp/request.bin` for IDE debugging.

## Architecture

### protoc-gen-open-models

Uses `bufbuild/protoplugin` framework. Entry point: `main.go` → `generator.GenerateAll()`.

**Two-pass generation** (`pkg/generator/generator.go`):
1. First pass builds global registries: `allTypes`, `allEnums`, `projectFiles`
2. Second pass generates TypeScript per file

**Per-message output** (single `.ts` file with three types):
- `I`-prefixed interface (Literal) — `LiteralType.go`
- `T`-prefixed interface (Transport, proto wire names) — `TransportType.go`
- Model class extending `FieldNode` — `ModelType.go`

**Type resolution**: `typeResolver.go` handles Literal/Transport mappings; `modelTypeResolver.go` handles Model runtime types (`STRING`, `INT32`, `ARRAY<>`, `MAP<>`, `RECURSION<>`, etc.). Both use global registries to compute relative import paths.

**Source info**: `pkg/sourceinfo/` extracts comments and OpenAPI v3 annotations from proto descriptors.

### protoc-gen-om-jsonschema

Uses `google.golang.org/protobuf/compiler/protogen` framework. Everything in two files:
- `main.go` (~820 lines) — schema generation, field conversion, oneof handling
- `proto_parser.go` (~230 lines) — raw protobuf wire format parsing for min/max detection

**Plugin options**: `strict_any`, `strict_map`, `strict_oneof`, `exclude_packages`, `exclude_messages` (comma-separated, semicolon-separated values within options).

**Testing**: Tests compare generated output against golden files in `testfiles/` and `testfiles-strict/`. `testfiles/request.bin` is a captured CodeGeneratorRequest for direct testing without protoc.

## Key Conventions

- Dependencies are **vendored** (`vendor/` directories) — do not use `go mod download`
- Binary names must match `protoc-gen-<name>` exactly for protoc plugin discovery
- Proto field names: `GetJsonName()` for Literal/Model (camelCase), `GetName()` for Transport (snake_case)
- Fully-qualified type names start with `.` (e.g. `.google.protobuf.Timestamp`)
- Generated TS files include header: `// Code generated by furo protoc-gen-open-models. DO NOT EDIT.`
- JSON Schema files have no extension, flat directory, named by fully-qualified message name
- Proto3 zero-value limitation: unsigned types auto-get `minimum: 0`; use `format: "positive"` annotation for signed types
- Version is hardcoded as `const version` in each plugin's `main.go`

## Debugging with a DEBUGFILE

When investigating code generation bugs in `protoc-gen-open-models` or `protoc-gen-om-jsonschema`, ask the user to supply a DEBUGFILE (`/tmp/request.bin`). This file captures the exact `CodeGeneratorRequest` (all proto descriptors, options, and file list) and allows reproducing the issue without the user's full proto/protoc setup.

**Generating a DEBUGFILE**: Build `protoc-gen-debugfile` and add it alongside the normal plugins in the protoc invocation:
```bash
cd protoc-gen-debugfile && go build -o protoc-gen-debugfile .
```
```bash
protoc \
  --debugfile_out=. \
  --debugfile_opt=/tmp/request.bin \
  --open-models_out=./generated \
  -I./proto_dependencies -I./proto \
  $(find proto -iname "*.proto")
```

**Replaying a DEBUGFILE**:
```bash
cd protoc-gen-open-models
go build -o protoc-gen-open-models .
./protoc-gen-open-models --replay-request=/tmp/request.bin
```

For `protoc-gen-om-jsonschema`, the same approach works:
```bash
cd protoc-gen-om-jsonschema
go build -o protoc-gen-om-jsonschema .
./protoc-gen-om-jsonschema --replay-request=/tmp/request.bin
```

When a user reports a code generation bug, always ask them to provide this file — it is the fastest path to reproducing and fixing the issue.
